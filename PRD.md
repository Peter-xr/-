# AI海龟汤游戏 PRD

## 1. 产品概述
一个AI驱动的海龟汤游戏网站。玩家通过与AI主持人对话（提问推理）逐步还原故事真相。

## 2. 目标用户
- 喜欢推理游戏的玩家
- 想要轻松娱乐的上班族
- 对AI应用感兴趣的技术爱好者

## 3. 价值主张
- 通过AI主持人实时反馈，让每个玩家的问题都能继续推进推理
- 提供沉浸式的“揭晓真相”体验，强化娱乐与复盘价值
- 通过简单规则（是/否/无关）降低AI交互噪音，提高可玩性

## 4. 核心玩法与范围

### 4.1 MVP（必须做）
1. 游戏大厅
   - 展示游戏列表（不同难度/类型的海龟汤）
   - 点击“开始游戏”进入游戏页

2. 游戏页面
   - 展示汤面故事（题面）
   - 聊天输入框，可以输入问题
   - AI只回答"是"、"否"、"无关"
   - 显示对话历史
   - "查看汤底"按钮
   - "结束游戏"按钮

3. 汤底页面
   - 展示完整汤底
   - 展示玩家推理过程
   - "再来一局"按钮

### 4.2 后续可以做
- 用户登录系统
- 游戏记录保存
- 排行榜
- 自定义海龟汤
- 多人对战模式

## 5. 界面与交互设计要求
- 整体风格：神秘悬疑，深蓝色调
- 游戏大厅：卡片式布局，显示故事标题和难度
- 游戏页面：聊天界面风格，类似微信
- 汤底页面：揭晓真相的仪式感

## 6. 技术栈建议
- 前端：React + TypeScript + Tailwind CSS
- 后端（建议）：Node.js + Express
- AI：DeepSeek / Claude API
- 部署：Vercel

## 7. AI主持人规则与约束

### 7.1 MVP输出格式强约束
AI必须严格输出以下三类之一：
- `是`
- `否`
- `无关`

禁止输出其它内容（如解释、长段落、完整答案）。

### 7.2 防剧透策略（建议）
- 系统提示词中明确：不得泄露完整汤底
- 可在服务端/前端加入规则化校验（例如模型输出不在三类之一则重试或替换分类兜底）

## 8. 功能验收标准（Acceptance Criteria）

### 8.1 游戏大厅
- 可渲染至少 3 个海龟汤条目（可用内置题库）
- 每个条目可点击进入游戏页

### 8.2 游戏页面
- 展示题面
- 玩家提问后：
  - AI返回必须属于 `是` / `否` / `无关` 三类之一
  - 对话历史实时更新
- `查看汤底` 按钮可进入汤底页面或展示汤底
- `结束游戏` 按钮可正确回到大厅且清理当前会话状态

### 8.3 汤底页面
- 正确展示完整汤底
- 展示推理过程（对话历史至少包含玩家与AI）
- `再来一局` 可开始新一轮（下一题或同题重置，需在实现说明中明确）

## 9. 非功能性需求（NFR）
- 移动端适配：小屏不溢出、不遮挡、输入可用
- 稳定性：AI请求失败时有可理解提示，不影响页面可操作性
- 安全性：API Key 禁止硬编码；建议使用后端代理或严格使用最小权限与环境变量方案
- 性能：对话列表渲染不卡顿（必要时对消息数量做基础限制）

## 10. 风险与对策
- 风险：AI不稳定输出导致出现解释或泄露答案
  - 对策：强提示 + 输出校验/重试/分类兜底
- 风险：纯前端直接调用AI导致密钥暴露
  - 对策：后端代理（推荐）或前端最小暴露与严格安全策略
- 风险：移动端输入体验差
  - 对策：优先使用响应式布局，确保输入区域可见与滚动策略合理

## 11. 待确认问题（Open Questions）
- AI供应商最终选型：DeepSeek 还是 Claude？（决定接口格式与安全策略）
- 是否需要真正多页路由（大厅/游戏/汤底）还是先用单页状态切换？
- “再来一局”策略：下一题随机还是同题重置？
- “结束游戏”后是否保留历史用于复盘还是直接清空？
